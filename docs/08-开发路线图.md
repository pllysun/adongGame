# 08 · 开发路线图

原则：**每个里程碑结束时游戏都能跑、能玩、能存档**。垂直切片优先，宽度后补。

## 里程碑总览

| 阶段 | 主题 | 验收标准（玩家视角） |
|------|------|---------------------|
| M0 | 骨架 | 项目跑起来：时间流动、变速暂停、空宗门界面、存读档 |
| M1 | 弟子与修炼 | 弟子会修炼、突破、老死；名册和详情页可看 |
| M2 | 宗门经营 | 资源经济转起来；三件套设施；第一届招收大会 |
| M3 | 事件系统 | 事件弹窗与事件流上线；首批 ~20 个事件 |
| M4 | 战斗 | 妖兽袭山可打；战前点将 + 战报文学 |
| M5 | 性格与关系网 | 弟子有性格、会社交、出人事事件；编年史上线 |
| M6 | 内容与平衡 | 事件 60+/功法 40+/丹药 20+；无头模拟回归达标 |
| M7 | 立绘与包装 | 纸娃娃正式素材；音效；Tauri 打包双平台 |

**MVP = M0~M4**：跑通"创宗→招人→修炼→突破→危机→战斗"完整循环，
此时即可回答"这游戏好不好玩"，并决定 M5+ 的投入与方向。

## 各里程碑要点

### M0 · 骨架（地基，宁慢勿糙）
- 脚手架：Vite + Vue3 + TS strict + Pinia + Tailwind + ESLint（含 engine 禁入规则）+ vitest
- 引擎核心：时钟/tick 调度、种子 RNG、命令与快照门面、存档 v1 + 迁移框架
- UI 壳：主布局三栏、顶栏时间控制、空白主视图路由
- 无头运行器能跑（空世界 1000 年不崩）

### M1 · 弟子与修炼
- 弟子生成（姓名/七品资质/灵根/悟性心性气运/年龄），程序化占位立绘
- 修为日结算、瓶颈、突破判定全公式、失败结果表、寿元与老死
- 屏幕：弟子名册（排序筛选）、弟子详情（数值版）
- 无头验证①：单弟子万次突破模拟 ≈ 矩阵理论值；500 年宗门境界分布合理

### M2 · 宗门经营
- 资源月结算（灵石/灵米/灵草）、月俸与待遇档
- 设施：聚灵阵/洞府/灵田（建造、升级、效果接入修炼公式）
- 开局流程（选址/祖师/命名）+ 招收大会（声望影响候选池）
- 宗门规则页 v1（晋升门槛、待遇、突破审批开关）

### M3 · 事件系统
- 事件引擎：条件 DSL、选角、权重抽取、事件链、autoPause
- 屏幕：事件弹窗、右侧事件流
- 首批事件 ~20：机遇 8 / 危机 6 / 人事 6（人事类先用简化触发，M5 接性格）
- 内容 schema 校验（zod）进 CI

### M4 · 战斗（MVP 收口）
- 战力模型、S 曲线胜负、节拍模拟、伤亡表、死亡保护阀
- 妖兽潮事件链全流程；战前点将屏幕；战报播放器
- 无头验证②：自动玩家跑 50 局 × 300 年，灭门率/首金丹年代进入目标区间
- **MVP 试玩评估点：邀请 1~2 人盲玩，记录"哪里想快进、哪里想暂停"**

### M5 · 性格与关系网
- 特质库 30+ 接入行为权重；关系边的形成/衰减/上限管理
- 人事事件接通性格触发（道侣/结仇/偷窃/叛逃/夺位）
- 心境系统全量（传导、心魔事件链）；协同系数入战斗
- 编年史屏幕 + 灭门/通关回顾生成

### M6 · 内容与平衡
- 内容冲刺：事件 60+、功法 40+、丹药 20+、节拍模板 30+、文案变体
- 丹房/藏经阁/演武场/护山大阵/静室全部上线
- 天劫系统、传功机制、声望飞轮调参
- 无头模拟 CI 化：每次提交 10 局快速回归，数值改动看报告说话

### M7 · 立绘与包装
- 纸娃娃正式图层素材（AI 生成→筛选→切层），表情 4 态，境界特效
- 音效/BGM（古风氛围），开场引导
- Tauri 打包：macOS + Windows（你的 Windows 机器做安装测试）
- 性能压测：200 弟子 × 10x 速度达标

## 风险清单

| 风险 | 缓解 |
|------|------|
| 数值失衡导致"必胜挂机"或"必死劝退" | 无头模拟从 M1 就开始，平衡问题早暴露 |
| 事件内容量不足，几局后重复感 | 选角系统+文案变体放大单事件复用率；M6 专门内容冲刺 |
| 关系网复杂度失控（性能或可理解性） | 边上限+社交池纪律；UI 只展示强关系 |
| 立绘素材风格不统一 | prompt 模板锁画风；占位方案保底（纯文字+色块也成立） |
| 范围蔓延（修真题材的诱惑：炼器/阵法/御兽/秘境地图……） | 一切新系统先进 docs/ 提案，过"三支柱"审判再排期 |

## 工作流约定

- 数值常量全部集中在 `engine/content/constants.ts`，改平衡 = 改一个文件 + 跑回归。
- 你（设计审阅者）的主战场是 `docs/` 和 `engine/content/`——所有玩法内容都是数据文件，
  审内容不需要读引擎代码。
- 每个里程碑收尾：可玩构建 + 更新文档与实际实现的差异（文档是活的）。
