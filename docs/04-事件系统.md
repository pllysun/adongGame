# 04 · 事件系统

事件系统是这个游戏的内容主干和故事出口。设计目标：**加一个事件 = 在数据文件里加一条记录**，
零引擎改动；事件能读世界任意状态、改世界任意状态、并串成多步剧情链。

## 0. 事件分级与分流（反"事件轰炸"）

为避免"每次都要点一堆事件"，事件按 **1-7 级**分流（对应七阶颜色：1 白 → 7 彩，越高越罕见、风险与收获越大）。

- **`level?` / `scope?` / `defaultOption?` 均为可选**——内容不写也能工作（关键：另有会话在并发加事件）。
  启发式默认：`pause:true → 4级`、`crisis → 3级`、其余 `2级`；`scope` 由 `personal`/`personnel` 推个人，否则宗门。
- **抽取权重 × 等级稀有度**：`levelRarity = [_,1,0.7,0.45,0.25,0.12,0.06,0.03][level]`，等级越高越难遇到。
- **拦截级别（`SectRules.interceptLevel`，默认 5 = 天品/金）**：
  - `level ≥ interceptLevel`（或 `pause:true`）→ **拦截队列**，弹窗打断 + 暂停时间（`world.queue`，阻塞 tick）。
  - `level < interceptLevel` → **收件箱**（`world.inbox`，不阻塞时间），玩家在「事件处理」页异步查看。
- **过期机制**：收件箱项 `expiryDay = day + INBOX_WINDOW_DAYS(600)`；到期 `expireInbox` 自动按
  `defaultOption`（缺省 = 最后一项，最保守/不作为）放弃处理。
- **指令**：`resolve`（队列与收件箱通用）、`dismissInbox`（立即按默认放弃）、`setRule interceptLevel`。
- 默认拦截级别下，强制弹窗从 ~6/年降到 ~0.5/年（无头实测），收件箱受过期清理约束不堆积。
- 注：等级数字↔颜色按"1-7 对应七色"——**4 级 = 地品（紫）**，5 级才是天品（金）。

## 1. 数据结构

```ts
interface GameEvent {
  id: string
  category: 'opportunity' | 'crisis' | 'personnel'   // 机遇 / 危机 / 人事
  trigger: Condition          // 声明式条件（见 §2）
  weight: number              // 基础权重，可被条件内的系数调制
  cooldownYears?: number      // 冷却
  once?: boolean              // 全局唯一（如转世重修者）
  autoPause?: boolean         // 弹出时是否自动暂停（重大事件 true）
  cast: CastSlot[]            // 角色槽：从世界里选演员（见 §3）
  text: string                // 模板文本，支持 {slot.name} 插值
  options: EventOption[]      // 玩家选项（人事类可为空 = 纯通告）
}

interface EventOption {
  text: string
  requirements?: Condition    // 不满足则置灰并显示原因（"需灵石 5000"）
  effects: Effect[]
  outcomes?: WeightedOutcome[] // 概率分支：同一选择可能有不同结果
}
```

## 2. 条件与效果 DSL（声明式，可序列化）

```ts
// 条件：对世界状态的查询，可组合
type Condition =
  | { all: Condition[] } | { any: Condition[] } | { not: Condition }
  | { stat: StatPath, gte?: number, lte?: number }      // { stat:'sect.reputation', gte:200 }
  | { exists: DiscipleQuery }                            // 存在满足条件的弟子
  | { worldFlag: string, eq: unknown }                   // 世界状态变量
  | { chance: number }                                   // 纯随机门

// 弟子查询：既用于条件，也用于角色槽选角
type DiscipleQuery = {
  realm?: RealmRange, aptitude?: RarityRange, trait?: TraitTag,
  relationWith?: { slot: string, type: RelationType, valueGte?: number },
  sortBy?: 'luck' | 'random' | 'power', ...
}

// 效果：对世界状态的修改
type Effect =
  | { addResource: { type: ResourceType, amount: number } }
  | { modifyMood: { slot: string, delta: number } }
  | { modifyRelation: { a: string, b: string, type?: RelationType, delta: number } }
  | { addTrait: { slot: string, trait: TraitTag } }
  | { grantItem: { slot?: string, itemRoll: LootTable } }   // 功法/丹药/法宝掉落
  | { spawnCombat: CombatSpec }                              // 移交战斗系统，战果回填事件链
  | { scheduleEvent: { id: string, delayDays: [min, max] } } // 事件链的链接器
  | { setWorldFlag: { key: string, value: unknown } }
  | { injure | kill | exile | recruit ... }
```

## 3. 角色槽（cast）——让事件文本有名有姓

事件触发时，引擎按 `cast` 声明从世界中**选角**：

```ts
// 例：弟子争执事件需要两个有矛盾苗头的同门
cast: [
  { slot: 'a', query: { trait: '傲慢', realm: { lte: '筑基' } } },
  { slot: 'b', query: { relationWith: { slot: 'a', type: '同门' }, sortBy: 'random' } },
]
text: "{a.name}与{b.name}在演武场切磋时言语相讥，竟至拔剑相向……"
```

选角失败（找不到符合条件的弟子）= 事件本次不触发。**这是涌现感的关键**：
同一事件库，因为宗门人员构成不同，每局触发的故事完全不同。

## 4. 调度模型

- 每**旬**（10 天）做一次事件抽取：从满足 trigger 的事件池里按权重抽 0~1 个
  （泊松节流，目标密度：平均每游戏年 4~8 个弹出事件，可按游戏阶段调）。
- **人事类事件**额外由行为系统直接触发（弟子行为选择的副产物），不占抽取预算。
- 个人气运参与权重：涉及具体弟子的机遇事件，选角时 `sortBy: 'luck'` 加权。
- 重大事件（`autoPause`）弹窗 + 暂停；小事件只进右侧事件流，不打断。

## 5. 事件链

`scheduleEvent` + `worldFlag` 组合即可表达多步剧情。例：

```
妖兽潮链：妖兽异动(通告) → [1~3月后] 兽群压境(选项：迎击/死守/疏散)
        → spawnCombat → 战后分支(大胜→声望+材料 / 惨胜→葬礼链 / 溃败→设施损毁+被掳链)
```

## 6. 首发事件库示例（按类别各举数例，首发目标 60+ 条）

### 机遇类
| 事件 | 触发要点 | 选项要点 |
|------|---------|---------|
| 秘境开启 | 周期+随机，品阶随世界年数提升 | 派队探索（选人界面）→ 战斗+寻宝结算 → 功法/灵草/材料 |
| 坊市拍卖 | 周期 | 竞拍功法/丹方/法宝（灵石博弈，有托儿陷阱小概率） |
| 散修来投 | 声望 ≥ 阈值 | 收/拒/考验后再收（考验触发小事件揭示性格） |
| 天材地宝出世 | 低概率，全世界争夺 | 派高手抢夺（强敌战斗）/ 坐视（敌宗实力+） |
| 转世重修者 | once，气运链 | 仙品资质婴儿入门，带全局事件链与觊觎者 |

### 危机类
| 事件 | 触发要点 | 选项要点 |
|------|---------|---------|
| 妖兽潮 | 妖兽活跃度积累 | 见 §5 链式示例 |
| 敌宗寻衅 | 敌意值 ≥ 阈值 | 赔罪（声望-）/ 斗法（双方各出三人）/ 开战（宗门战链） |
| 弟子被掳 | 外出弟子+魔道活跃 | 营救（战斗）/ 赎金 / 放弃（全宗心境-，侠义弟子可能私自去救） |
| 心魔劫 | 心境 ≤ -60 的弟子 | 长老护持（占用人力）/ 静室闭关 / 放任（入魔概率高） |
| 灵脉震荡 | 低概率 | 聚灵阵效率临时砍半，抢修消耗材料 |

### 人事类（行为系统直接触发，涌现的主力）
| 事件 | 触发要点 |
|------|---------|
| 私定道侣 | 倾慕双向 ≥ 阈值；若门规禁止 → 私奔检定 |
| 演武结仇 | 切磋 + 傲慢/记仇特质 → 仇怨边建立 |
| 偷窃灵石 | 贪婪 + 欠俸 → 执法堂事件（玩家裁决：逐出/杖责/宽恕，各有关系涟漪） |
| 夺位之争 | 掌门坐化未立继任 + 多名野心长老 |
| 叛逃投敌 | 仇怨+待遇不满累积 → 带走功法（藏经阁等级影响可带走的品阶） |
| 资质觉醒 | 大器晚成弟子经历生死事件后触发 |

## 7. 文本生产规范

- 模板文本写**短**（2~4 句），重要的是插值后读起来像那个弟子的事。
- 同一事件配 2~3 套文案变体（按性格标签选用），降低重复感。
- 所有事件文本进 `data/events/*.ts`，按类别分文件，review 即审稿。
